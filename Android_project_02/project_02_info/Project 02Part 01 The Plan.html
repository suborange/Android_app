Goal and Learning Outcomes

The major goal of this assignment is to plan out the android application you will be building over the remainder of the semester.  This project may be completed alone or working with a teammate. 

Walkthrough of an example application

For this assignment you will decide on what type of application you will be creating and come up with 4 of the initial use cases for the assignment. In addition to the use cases I want students to think about what data will be stored and how those data will be used.  This document WILL change throughout the development of this project.  The goal here is to get started thinking about the project.  With the project in the back of your mind the Android assignments will be a better help in creating the application. 

I have provided outlines for the first few use cases in this document. 

This assignment is mostly concerned with completing this document and turning it in.

 

Learning outcomes addressed

CLO1: Object Oriented Software Design

B.   Analysis and application of abstraction

C.   Comprehend and discuss software life cycles

D.   Comprehend and discuss different software development process models

E.    Apply use case modeling techniques.

 

CLO3 Knowledge of the Android Platform

A.    Develop and appraise an Android app using Android Studio.

B.   Apply object-oriented design to Android development

 

What we will do

 

Create a copy of this document

 

Any field that is surrounded by [ ] should be filled in with your information, for example:
For me [Your name] would become Drew A. Clinkenbeard

Two example test cases are shown below. 

Two use cases are shown below, {Create Account}, is a main use case, and {Bad username} is an alternate flow to the main use case. Use this document as a guide.

Each use case should incorporate several primary and secondary flows. As an example:

{Create Account}

  1. User clicks the 'create account' button
  2. Application displays the create account screen
  3. user types in desired user name
  4. user types in the desired password
  5. user clicks the create account button
  6. user is returned to the landing page
  7. a toast message is displayed indicating the successful creation of an account
  8. use case ends

Alternate: {Bad username}

{user enters a username that is already taken}
Starting at 3 from {create account}

  1. User enters a username that is already in use
  2. Application displays a message "Username already in use"
  3. User enters the same username again
  4. Application returns to the landing page
  5. Application displays message "Too many bad attempts"
  6. Use case ends.

These use case may be used in your application.

The numbering for all the flows should start at 1.  The alternate flows may continue from where they diverge from the main flow or start over at 1. That is if there are 8 steps in the main flow, the alternate flow should NOT start at 9.  The alternate flow should state where in the main flow it begins (such as "at 3 of {Create Account}") and then the alternate flow will either start at 3 (the point where it diverges from the main flow) OR the alternate flow will start at 1.  Either way is fine so long as it is consistent.  

Be sure to update the table of contents when you are finished

NOTE: these use cases will likely change as the application is developed. This is normal and to be expected.
